# recreation_accessibility

Воспроизводимая Python-реализация алгоритма оценки обеспеченности
рекреационными объектами НИР «Разработка метода оценки обеспеченности
рекреационными объектами и формирование рекомендаций для их размещения»
(Антонова А. А., ИТМО, 2026).

## Алгоритм

### Качество объектов 

| Коэффициент | Источник | AHP-вес |
|---|---|---|
| `k_zone` | Функциональное зонирование | 0.263 |
| `k_ndvi` | Средний NDVI по полигону | 0.558 |
| `k_water` | Расстояние до водоёма | 0.088 |
| `k_road` | Расстояние до магистрали | 0.091 |

Скорректированная вместимость:
`C'_j = capacity_j × α_внеграницы × (1 + Σ w_k · (k_j − 1))`

### Обеспеченность (Huff-2SFCA)

Три уровня обслуживания (local 400 м / district 1500 м / city 6000 м)
с гауссовым затуханием `exp(−β·d)` и весами 0.5 / 0.3 / 0.2.

`A_h_i = Σ_level w_level · A_{level}_i`

### Стресс-индекс

```
S_i = max(0, 1 − A_h_i / A_m)
S_A = Σ (P_i · S_i) / Σ P_i
```

## Запуск на собственных данных

### Шаг 1 — подготовить папку `data/`

| Файл | Формат | Обяз. | Требования |
|---|---|---|---|
| `boundary.gpkg` | GeoPackage (Polygon) | да | колонки `NAME`, `ADMIN_LVL`, `OSM_ID` |
| `recreation.gpkg` | GeoPackage (Polygon) | да | только геометрия; площадь/вместимость считаются автоматически |
| `building.gpkg` | GeoPackage (Polygon) | да | см. раздел «Атрибуты зданий» ниже |
| `water.gpkg` | GeoPackage (Polygon/Line) | да | только геометрия |
| `Genplan23.gpkg` | GeoPackage (Polygon) | нет | колонка `label`/`ZONE` с именем зоны |
| `NDVI.tif` | GeoTIFF (Float32) | нет | значения в диапазоне [−1; 1]; Landsat DN×0.0001 исправляются автоматически |
| `AIS.csv` | CSV (sep=`;`, UTF-8) | нет | колонки `address`, `number_floors_max` |

**CRS**: все векторные слои должны иметь заданную CRS.
Если `crs: ""` в `config.yaml` (рекомендуется), система координат определяется
автоматически из `boundary.gpkg`: для географических CRS вычисляется UTM-зона
по центроиду, для проецированных CRS берётся напрямую.
Остальные слои автоматически перепроецируются в выбранную СК.

---

### Атрибуты зданий (`building.gpkg`)

Код читает этажность и жилую принадлежность из нескольких источников
по принципу каскадного приоритета.

#### Фильтр жилых зданий (приоритет сверху вниз)

| Колонка | Тип | Значение «жилое» |
|---|---|---|
| `is_living` | int / float / bool | `1` / `True` / `1.0`; `NaN` и `0` = нежилое |
| `BUILDING` или `building` | str (OSM-тег) | `residential`, `apartments`, `house`, `detached`, `semidetached_house`, `terrace`, `dormitory`, `allotment_house` |
| *(нет ни одной)* | — | все здания считаются жилыми |

> Если в OSM-выгрузке тег `building=yes` (общий, без типа) и нет `is_living`,
> такое здание попадёт в нежилые. Создайте колонку `is_living` вручную или дополните выборку через `[building~"residential|apartments|house"]`.

#### Этажность (каскад)

| Колонка | Описание |
|---|---|
| `building:levels` | OSM-тег; приоритет 1 |
| `B_LEVELS` | Поле геоданных (напр., Росреестр); приоритет 2 |
| `height` | Высота здания в метрах; делится на 3; приоритет 3 |
| *(не задана)* | Медиана соседей в радиусе `floor_search_radius` м → глобальная медиана → fallback 2 эт. |

### Шаг 2 — настроить `config.yaml`

Минимальные изменения для новой территории:

```yaml
crs: "EPSG:32637"          # выбрать UTM-зону под свою территорию
                            # СПб/Кронштадт: EPSG:32636
                            # Москва:        EPSG:32637

paths:
  boundary:   "data/boundary.gpkg"   # путь к файлу
  recreation: "data/recreation.gpkg"
  building:   "data/building.gpkg"
  water:      "data/water.gpkg"
  zoning:     "data/Genplan23.gpkg"  # "" если нет
  ndvi:       "data/NDVI.tif"        # "" если нет
  gkh_csv:    "data/AIS.csv"         # "" если нет

population:
  sqm_per_person: 28        # норматив жилой площади на чел. (СПб = 28 м²)

capacity_per_ha: 100        # нормативная вместимость рекреации, чел/га
```

### Шаг 3 — структура `boundary.gpkg`

Файл описывает административную иерархию территории.
Минимально необходимые колонки:

| Колонка | Тип | Описание |
|---|---|---|
| `NAME` | str | Отображаемое имя (для меню выбора) |
| `ADMIN_LVL` | str / int | Уровень иерархии (`4`, `5`, `8` — или любые числа) |
| `OSM_ID` | int | Уникальный идентификатор |

При запросе `python run.py` каскад перебирает уровни в порядке возрастания.
Если иерархии нет (один уровень, `ADMIN_LVL` одинаковый для всех) —
интерактив покажет список сразу всех объектов.

Подготовка минимального `boundary.gpkg` из произвольного полигона:
```python
import geopandas as gpd
from shapely.geometry import shape

gdf = gpd.GeoDataFrame({
    "NAME": ["Мой район"],
    "ADMIN_LVL": ["8"],
    "OSM_ID": [1],
}, geometry=[<ваш_полигон>], crs="EPSG:4326")
gdf.to_file("data/boundary.gpkg", driver="GPKG")
```

## Сценарный анализ

Все сценарии строятся поверх **baseline**: запустите его первым,
чтобы зафиксировать пороги классификации Q25/Q75.

### Сценарий 1 — инфраструктурный (`run_infrastructure_scenario`)

**Что умеет:**
- добавить новые рекреационные объекты (`new_recr`)
- изменить коэффициенты качества существующих объектов (`modify`)

#### Формат `new_recr`

GeoDataFrame c полигонами. Обязательных атрибутов нет — всё вычисляется автоматически:

| Колонка | Тип | По умолчанию (если не задать) |
|---|---|---|
| `geometry` | Polygon | *обязательна* |
| `service_level` | str | определяется по площади: < 3 га = local, 3–15 га = district, > 15 га = city |
| `area_m2` | float | `geometry.area` |
| `capacity` | float | `area_m2 / 10 000 × capacity_per_ha` |
| `k_zone` | float | 1.20 (наилучшее зонирование) |
| `k_ndvi` | float | 1.30 (наилучший NDVI) |
| `k_water` | float | 1.30 (у воды) |
| `k_road` | float | 1.00 (нейтрально) |
| `within_boundary` | bool | `True` |

Минимальный пример — один объект по заданным координатам:

#### Параметр `modify` — изменение существующих объектов

`modify` принимает словарь `{индекс: {поле: значение}}`.

**Что такое индекс?** Это метка строки в GeoDataFrame `rec0` (не порядковый номер).
После каждого `load_all` + `apply_quality` индексы одни и те же — определяются
порядком объектов в `recreation.gpkg`.


Доступные для изменения поля и допустимые значения:

| Поле | Диапазон | Смысл |
|---|---|---|
| `k_zone` | 0.60 – 1.20 | функциональное зонирование |
| `k_ndvi` | 0.50 – 1.30 | озеленение |
| `k_water` | 1.00 – 1.30 | близость к воде |
| `k_road` | 0.50 – 1.00 | удалённость от магистралей |

---

### Сценарий 2 — демографический (`run_demographic_scenario`)

**Что умеет:**
- добавить новые жилые здания (`new_buildings`)
- масштабировать население существующих зданий (`growth_pct`)

#### Параметр `growth_pct`

Умножает поле `population` всех существующих зданий на `(1 + growth_pct/100)`.

#### Формат `new_buildings`

GeoDataFrame c полигонами зданий. Население вычисляется по приоритету:

| Вариант | Что нужно в файле | Поведение |
|---|---|---|
| **А — из данных** | колонка `population` | берётся напрямую |
| **Б — из геометрии** | колонка `floors` + полигон | `population = площадь × floors × living_ratio / sqm_per_person` |
| **В — минимальный** | только полигон | `floors = медиана существующих`, далее как Б |
```
